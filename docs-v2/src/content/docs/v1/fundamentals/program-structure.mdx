The Anchor framework uses [Rust macros](https://rust-book.cs.brown.edu/ch20-05-macros.html) to reduce boilerplate and enforce the security checks every Solana program needs. Four macros form the backbone of any Anchor program.

- [`declare_id!(){:rs}`](#declare_id) sets the program's on-chain address.
- [`#[program]{:rs}`](#program) annotates the module containing the program's instruction handlers.
- [`#[derive(Accounts)]{:rs}`](#deriveaccounts) declares the accounts an instruction requires.
- [`#[account]{:rs}`](#account) defines a custom account data type owned by the program.

<Code code={lib} lang="rust" title="lib.rs" meta="{3} collapse={4-27}" />

By default, the program ID is the public key of the keypair at `/target/deploy/<program_name>-keypair.json{:file}`. To rewrite the `declare_id!(){:rs}` string to match the on-disk keypair, run `$ <blue>anchor</blue> keys sync`.

<Callout variant="tip" title="When to run anchor keys sync">
  Run `$ <blue>anchor</blue> keys sync` after cloning a repository. The committed program ID will
  not match the keypair generated by `$ <blue>anchor</blue> build` on your machine, so syncing
  prevents a mismatch between the declared ID and the deployed binary.
</Callout>

### Instruction context

The first parameter of every instruction handler is a [`Context<T>{:rs}`](https://docs.rs/anchor-lang/latest/anchor_lang/context/struct.Context.html), where `T{:rs}` is a struct that implements the [`Accounts{:rs}`](https://docs.rs/anchor-lang/latest/anchor_lang/trait.Accounts.html) trait. The context exposes everything the handler needs from the runtime beyond its own arguments:

```rust showLineNumbers=false
pub struct Context<'a, 'b, 'c, 'info, T: Bumps> {
    /// Currently executing program id.
    pub program_id: &'a Pubkey,
    /// Deserialized accounts.
    pub accounts: &'b mut T,
    /// Remaining accounts given but not deserialized or validated.
    /// Be very careful when using this directly.
    pub remaining_accounts: &'c [AccountInfo<'info>],
    /// Bump seeds found during constraint validation.
    pub bumps: T::Bumps,
}
```

Access each field on the `ctx{:rs}` parameter using dot notation:

- `ctx.accounts{:rs}` contains the deserialized, validated accounts declared by `T{:rs}`.
- `ctx.program_id{:rs}` is the public key of the currently executing program.
- `ctx.remaining_accounts{:rs}` holds any accounts passed beyond the `Accounts{:rs}` struct. These are neither deserialized nor validated, so use with caution.
- `ctx.bumps{:rs}` exposes bump seeds discovered during PDA constraint validation.

Any additional parameters on the handler function become instruction arguments supplied by the client at invocation time. In the example, `data: u64{:rs}` is an argument:

Each field maps to one account the instruction requires. Field names are arbitrary but should describe the role the account plays in the instruction.

### Account validation

Account validation is the first line of defense against malicious input. Anchor enforces two complementary mechanisms, typically used together.

- **[Account constraints](/docs/v1/reference/account-constraints/)** declared in the `#[account(...)]{:rs}` attribute enforce runtime conditions on each account. Common constraints include `init{:rs}`, `mut{:rs}`, `has_one{:rs}`, `seeds{:rs}`, and `constraint{:rs}`.
- **[Account types](/docs/v1/programs/account-types/)** specified in the field's type enforce static expectations about what kind of account the client must pass. Examples include `Account<T>{:rs}`, `Signer{:rs}`, `Program<T>{:rs}`, and `UncheckedAccount{:rs}`.

Once validation succeeds, the handler accesses accounts through `ctx<cyan>.</cyan>accounts<cyan>.</cyan><dim><field_name></dim>`.

Accounts created with this type are owned by the program identified in `declare_id!(){:rs}`. Anchor writes an 8-byte discriminator to the front of the account at initialization and checks it on every read, and serializes and deserializes the account's data through Borsh.

### Account discriminator

The discriminator is the first 8 bytes of `SHA256("account:<StructName>"){:rs}`, stored as the leading 8 bytes of every account created with this type ([source](https://github.com/otter-sec/anchor/blob/62865c636aecc6974fc9cfebfc6cf08ca4f0bb72/lang/attribute/account/src/lib.rs#L113-L130)). It serves two purposes.

- On **initialization**, Anchor writes the discriminator as the first 8 bytes of the account's data.
- On **deserialization**, Anchor compares the stored discriminator against the one expected by the target type. A mismatch fails the instruction before the handler runs, preventing the client from substituting an account of the wrong type.

<Callout variant="important" title="Reserving discriminator space">
  Every `init{:rs}` constraint must allocate 8 extra bytes of space for the discriminator in addition to the account's own fields. In the example below, `space = 8 + 8{:rs}` covers 8 bytes for the discriminator plus 8 bytes for the `data: u64{:rs}` field.

```rust showLineNumbers=false
#[account(init, payer = signer, space = 8 + 8)]
pub new_account: Account<'info, NewAccount>,
```

</Callout>
