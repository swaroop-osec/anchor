## Background: how Anchor picks the right handler

Every instruction Anchor dispatches starts with a **discriminator**:
the leading bytes of the instruction data that tell the program which
handler to run. By default the `#[program]` macro hashes
`instruction:<handler_name>` and uses the first 8 bytes; the
macro-emitted entrypoint reads those 8 bytes, matches them in a table,
and calls the matched handler. Two levers in this page touch that
flow:

- **`#[discrim = N]`** swaps the 8-byte hash for a single byte you
  choose, but the macro still owns the entrypoint and the dispatch
  match. Cheaper on the wire, no other change.
- **Custom entrypoint** replaces the macro's entrypoint with your own
  asm or Rust function. You read the disc byte yourself, fast-path the
  one or two instructions that matter, and hand everything else back
  to the macro-emitted dispatcher (exposed as `__anchor_dispatch`).
  Pairs naturally with `#[discrim]` because reading one byte is
  cheaper than reading and matching eight.

The rest of the page covers when each is worth it and how to wire
them up.

## When to reach for this

You have a program where one instruction dominates the call volume
(an oracle `update`, a price tick, a transfer between two known
accounts) and that instruction does very little work. The dispatcher
overhead itself (deserializing the instruction discriminator,
walking accounts, calling `try_accounts`) becomes a meaningful
fraction of the budget.

There are two related but distinct levers, and they layer on top of
each other:

| Lever | What it does | Needs `no-entrypoint` feature? | Example |
|---|---|---|---|
| **`#[discrim = N]` attribute** | Pins each handler to a one-byte discriminator. Macro still emits the entrypoint and dispatches; you just change the on-wire disc bytes. | No | [`tests-v2/programs/spl`](https://github.com/otter-sec/anchor/tree/anchor-next/tests-v2/programs/spl) |
| **Custom entrypoint** | Replaces the macro-emitted entrypoint with hand-written sBPF assembly or raw Rust. You read the disc, fast-path one or more instructions inline, fall through to `__anchor_dispatch` for everything else. | Yes | [`bench/programs/prop-amm/anchor-v2`](https://github.com/otter-sec/anchor/tree/anchor-next/bench/programs/prop-amm/anchor-v2) (asm) |

The CU win on the runnable example (prop-amm `update`) is **26 CU**,
roughly **50×** over v1's 1,310 CU. The default-v2 macro-dispatched
path isn't measured separately (the bench builds v2 with the asm
entrypoint enabled); use `prop-amm/initialize` at 1,375 CU as a
stand-in for what the macro path would cost on a handler of similar
shape.

## `#[discrim = N]` alone

This is the lightweight option. You don't replace anything, you just
override the macro's default 8-byte SHA-256 discriminator with a single
byte of your choosing. The macro still emits the entrypoint, allocator,
panic handler, and dispatch table; they just match on `N` instead of
the hashed bytes.

```rust title="src/lib.rs"
use anchor_lang_v2::prelude::*;

#[program]
pub mod token_demo {
    use super::*;

    #[discrim = 0]
    pub fn init_mint(_ctx: &mut Context<InitMint>) -> Result<()> {
        Ok(())
    }

    #[discrim = 1]
    pub fn init_token_account(_ctx: &mut Context<InitTokenAccount>) -> Result<()> {
        Ok(())
    }
}
```

`Cargo.toml` is the standard v2 template, with `no-entrypoint`
staying out of `default`. `N` is an integer literal in `u8` range. The
executable `#[program]` mode is all-or-nothing: if any handler uses
`#[discrim = N]`, every handler must use a one-byte discriminator, and
duplicates are rejected at compile time. The
[`tests-v2/programs/spl`](https://github.com/otter-sec/anchor/tree/anchor-next/tests-v2/programs/spl)
program uses this pattern with `#[discrim = 0..22]` across its full
SPL surface for a complete reference.

## Custom entrypoint (asm or Rust)

When `#[discrim]` alone isn't enough and you want to skip Anchor's
dispatcher entirely on a hot path, you replace the entrypoint and
dispatch on the disc byte yourself, falling back to
`__anchor_dispatch` for any disc you don't fast-path. This needs two
changes on top of the standard v2 template:

1. Flip `no-entrypoint` into `default` features.
2. Reinstate the allocator + panic handler the feature suppresses,
   then provide your own `entry` fn.

```toml title="Cargo.toml"
[features]
default = ["no-entrypoint", "no-log-ix-name"]
no-entrypoint = []
cpi = ["no-entrypoint"]
no-idl = []
no-log-ix-name = []

[dependencies]
anchor-lang-v2 = { git = "https://github.com/otter-sec/anchor", branch = "anchor-next", default-features = false, features = ["alloc"] }
```

The only diff vs. a standard v2 program is the `default = [...]` line.
Adding `"no-entrypoint"` triggers the macro's `#[cfg(feature =
"no-entrypoint")]` checks at build time: the entrypoint symbol +
allocator + panic handler stop being emitted, and `__anchor_dispatch`
becomes the unmangled symbol your code can call.

```rust title="src/lib.rs"
use anchor_lang_v2::prelude::*;

declare_id!("...");

#[program]
pub mod oracle {
    use super::*;

    /// Unreachable: the custom entrypoint handles `discrim=0` before
    /// this arm is reached. Declared so the macro still emits
    /// `instruction::Update` + `accounts::Update` for clients and IDL.
    #[discrim = 0]
    #[allow(unused_variables)]
    pub fn update(ctx: &mut Context<Update>, new_price: u64) -> Result<()> {
        unsafe { core::hint::unreachable_unchecked() }
    }

    #[discrim = 1]
    pub fn initialize(ctx: &mut Context<Initialize>) -> Result<()> {
        Ok(())
    }
}

// Reinstate the allocator + panic handler that `no-entrypoint`
// suppresses, then provide your own entrypoint.
#[cfg(target_os = "solana")]
anchor_lang_v2::pinocchio::default_allocator!();
#[cfg(target_os = "solana")]
anchor_lang_v2::pinocchio::default_panic_handler!();
```

From here you have two implementation choices for the entrypoint
itself:

### Asm

Hand-written sBPF assembly with hardcoded offsets against agave's
account serialization layout. Lowest CU floor (~26 in prop-amm).

The minimal shape mirrors the Rust example below: read disc byte at
`r2`, short-circuit one discriminator, tail-call `__anchor_dispatch`
for everything else:

```text title="src/asm/entrypoint.s"
.globl entrypoint
entrypoint:
    ldxb r3, [r2+0]
    jne r3, 0, dispatch
    mov64 r0, 0
    exit
dispatch:
    call __anchor_dispatch
    exit
```

The realistic example with full per-handler checks (account count,
data length, signer, address compare) lives at
[`bench/programs/prop-amm/anchor-v2/src/asm/entrypoint.s`](https://github.com/otter-sec/anchor/tree/anchor-next/bench/programs/prop-amm/anchor-v2/src/asm/entrypoint.s)
and lands `update` at 26 CU.

### Rust

A raw `extern "C"` entrypoint that branches on the disc byte and
either runs the fast path inline or forwards to `__anchor_dispatch`.
Higher CU floor than asm (pays the Rust prologue and any typed-view
costs you opt into) but the source reads like normal Rust and the
compiler catches misuse of typed APIs.

```rust title="src/lib.rs (continued)"
/// Raw SBF entrypoint, same signature the BPF loader hands to asm under
/// SIMD-0321. The `[ix_data_ptr - 8 .. ix_data_ptr]` slot holds the
/// `u64` ix-data length (set by agave's aligned serialization).
///
/// `__anchor_dispatch` is already declared at crate root by the
/// `#[program]` macro under `no-entrypoint`, so we call it via its
/// Rust path with no re-declaration needed.
#[cfg(target_os = "solana")]
#[no_mangle]
pub unsafe extern "C" fn entrypoint(input: *mut u8, ix_data_ptr: *const u8) -> u64 {
    let ix_data_len = *(ix_data_ptr.sub(8) as *const u64) as usize;

    // Empty ix data: nothing to dispatch on. Hand to `__anchor_dispatch`
    // so it can produce the typed `InstructionFallbackNotFound` error.
    if ix_data_len == 0 {
        return crate::__anchor_dispatch(input, ix_data_ptr);
    }

    let disc = *ix_data_ptr;
    match disc {
        // Fast path: `ping` is a no-op, so the inline branch just returns
        // success. Real fast paths would walk `input` for accounts +
        // read args past `ix_data_ptr.add(1)`.
        0 => 0,
        // Anything else: hand off to Anchor's typed dispatcher.
        _ => crate::__anchor_dispatch(input, ix_data_ptr),
    }
}
```

`__anchor_dispatch` expects the same `(input, ix_data_ptr)` pair the
loader passes to your `entrypoint`, so the fallback is a single
forward call. No marshalling required. The pattern is sketched in
source comments at
[`lang-v2/derive/src/lib.rs:4064-4083`](https://github.com/otter-sec/anchor/tree/anchor-next/lang-v2/derive/src/lib.rs#L4064);
no end-to-end example program ships in this tree yet.

### Picking between them

| | Asm | Rust |
|---|---|---|
| CU floor on fast path | ~26 (prop-amm) | Higher; pays Rust prologue + any typed-view costs |
| Effort | High (hardcoded offsets) | Lower if you stick to pinocchio's typed API |
| Layout coupling | Pinned to agave serialization | Same on raw paths; abstracted when using `AccountView` |
| Compile-time errors on misuse | None | Yes (when using typed API) |
| In-tree example | [prop-amm](https://github.com/otter-sec/anchor/tree/anchor-next/bench/programs/prop-amm/anchor-v2) | None |

Reach for asm only when the CU delta is worth maintaining hand-written
offset tables against runtime upgrades. For most "skip the dispatcher"
wins, the Rust shape closes most of the gap with far less risk.

## Things this gives up

These apply to both asm and Rust custom entrypoints (the part that
matters is bypassing `__anchor_dispatch`):

- **Per-handler checks on the fast path.** Anchor normally runs
  discriminator dispatch, instruction-data length / argument decoding,
  account count, duplicate mutable account checks, owner/writable/signer
  flags, account data length, PDA derivation, and address comparisons
  via the generated `TryAccounts` impl. Whatever disc you fast-path
  skips all of that. Write the equivalent checks by hand or rely on
  pinocchio's typed `AccountView` API for the checks it actually covers.
- **Discriminator scheme is now part of your fast-path ABI.** The
  `#[discrim = N]` attribute keeps clients, IDL, and the custom
  entrypoint in sync. If you remove the attributes from every handler,
  the macro falls back to its default 8-byte discriminator scheme while
  your fast-path branch still reads one byte; if you remove only one,
  the executable program fails to compile because v2 requires
  all-or-nothing custom discriminators.
- **Coupling to `__anchor_dispatch`'s ABI.** Your fallback calls the
  macro-emitted dispatcher with `(input: *mut u8, ix_data_ptr: *const u8) -> u64`.
  Re-check this on Anchor upgrades; a macro change here is either a
  compile/link break or an ABI mismatch in your custom fallback.

### Asm-specific

- **Offsets are hardcoded against agave's serialization layout.** The
  prop-amm asm reads `oracle.data_len` at `[r1+88]`,
  `authority.is_signer` at `[r1+10393]`, etc. A runtime change to the
  layout requires updating the assembly. The compiler only sees numeric
  offsets; it will not warn you when the runtime layout moves.
- **Asm has no Rust safety checks.** The assembly itself jumps straight
  to checks → writes → exit. Before every byte/word load, validate the
  relevant instruction-data length, account count, account data length,
  and duplicate-account encoding. A bad raw offset can become a VM
  memory fault or a wrong in-bounds read/write; unsafe Rust called from
  the asm can additionally hit Rust UB if it creates invalid references.
  If the asm calls Rust helpers, keep those helpers non-allocating and
  non-panicking unless you intentionally support the reinstated allocator
  and panic handler on that path.

## What still works

- Clients generated from the IDL still call `instruction::Update` the
  same way. They serialize discriminator + args, the fast path reads
  them.
- CPI from another program hits this program's entrypoint the same
  way an external transaction does, so the custom dispatcher sees CPI
  calls too. The custom entrypoint branches on discriminator regardless
  of caller: fast-path discriminators take the inline path; everything
  else falls through to `__anchor_dispatch` and runs the typed handler.
- Tests written against the typed handler keep working as long as
  the LiteSVM/Surfpool harness exercises the real `.so` (which is
  what `bench/tests/` does).
