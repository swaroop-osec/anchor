## Overview

Anchor v2 is designed to be **fast by default**. A counter program built
with the standard `#[program]` macro is roughly **18×** smaller and **4×**
cheaper per instruction than its v1 counterpart, with no extra effort from
the developer. Most programs never need anything beyond the defaults.

The patterns documented here cover the remaining cases: hot paths
where a few hundred compute units matter, oracles where every byte of the
binary counts, or programs that need to express layouts the safe defaults
do not expose. Each pattern trades a guarantee for performance. Read the
trade-off page before reaching for one.

## Free wins (no opt-in required)

These are already on for every v2 program. Listed for awareness, not as
something to enable:

- **PDA bumps precomputed at macro time.** Literal seeds get folded into
  a `const` at compile time; the runtime skips `find_program_address`.
- **On-curve check skipped for program-owned PDAs.** A non-empty,
  program-owned account was created via signed CPI, which already
  validated the seeds. Saves ~1,000 CU per verify.
- **wincode instead of borsh on the wire.** Byte-identical to borsh for
  the shapes Anchor programs use, 3–10× cheaper to encode on SBF.
- **`CpiHandle` routes through `invoke_signed_unchecked`.** The Rust
  borrow checker rules out the aliasing UB at compile time, so the
  runtime borrow-state check is skipped.
- **`remaining_accounts()` is lazy.** First call walks the cursor;
  subsequent calls clone the cached vec. Instructions that never touch
  it pay zero overhead.
- **`Account<T>` is zero-copy by default.** Stored bytes are
  `[disc][repr(C) T]`, load is a pointer cast, exit is a no-op. See
  [Zero-copy layout](./zero-copy-layout) for the Pod
  constraints, the alignment-1 wrappers (`PodU64`, `PodBool`, ...), and
  the `PodVec` / `Slab` helpers for variable-length data.

## Escape hatches (opt-in)

| Pattern | When to reach for it | Trade-off |
|---|---|---|
| [Custom asm entrypoint](./custom-entrypoint) | One instruction is the entire hot path (oracle update, price tick). | Discriminator dispatch + every account/data check is now your problem. |
| [Raw `AccountView`](./raw-account-view) | A handler reads a single field and the typed wrapper's load is the dominant cost. | No discriminator/owner/borrow validation. You write the checks. |
| [Bytemuck events](./zero-copy-layout#bytemuck-events) | High-frequency events whose payload is fixed-size Pod. | No `Vec`/`String`/`Option` in the event body. |
| `--features const-rent` | Programs that call `create_account` often and have fixed `space`. Saves ~85 CU per call. | Rent formula is baked at compile time. A runtime formula change (e.g. SIMD-0194) requires rebuilding the program. |

## Examples

Worked programs live under
[`bench/programs/`](https://github.com/otter-sec/anchor/tree/anchor-next/bench/programs),
paired with v1 for head-to-head comparison. Binary sizes and CU
numbers are from the v2 variant.

| Program | Description | Binary | CU range | Bin ↓ | CU ↓ |
|---|---|---|---|---|---|
| [helloworld](https://github.com/otter-sec/anchor/tree/anchor-next/bench/programs/helloworld/anchor-v2) | Single-instruction counter | 6.9 KB | 1,383 | 18× | 4.2× |
| [prop-amm](https://github.com/otter-sec/anchor/tree/anchor-next/bench/programs/prop-amm/anchor-v2) | Oracle feed with asm fast-path | 9.2 KB | 26–1,383 | 15× | 3.1–50× |
| [vault](https://github.com/otter-sec/anchor/tree/anchor-next/bench/programs/vault/anchor-v2) | Single-depositor SOL vault | 5.9 KB | 403–1,910 | 18× | 3.0–6.1× |
| [nested](https://github.com/otter-sec/anchor/tree/anchor-next/bench/programs/nested/anchor-v2) | Shared-validation via `Nested<T>` | 13 KB | 476–2,748 | 12× | 7.2–10× |
| [multisig](https://github.com/otter-sec/anchor/tree/anchor-next/bench/programs/multisig/anchor-v2) | Four-instruction SOL multisig | 31 KB | 477–2,363 | 5.3× | 3.0–9.1× |

`prop-amm`'s 26 CU low end is the asm fast path on `update`. Every
other entry uses the default macro-dispatched path, no escape hatch.
